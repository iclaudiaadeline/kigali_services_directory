## 1. Firestore Database Structure

### Overview
The application uses Firebase Cloud Firestore as its primary database, organized into two main collections: `users` and `listings`. This structure provides clear separation of concerns while maintaining efficient query performance.

### Collections and Document Schema

#### 1.1 Users Collection (`users`)
**Purpose**: Store user profile information for authenticated users.

**Document ID**: User's Firebase Authentication UID (auto-generated)

**Fields**:
```json
{
  "uid": "string - Firebase Auth UID",
  "email": "string - User's email address",
  "displayName": "string - User's full name",
  "emailVerified": "boolean - Email verification status",
  "createdAt": "string (ISO 8601) - Account creation timestamp"
}
```

**Security Rules**:
- Read: Any authenticated user
- Write: Only the user themselves (UID match)

**Rationale**: User profiles are created automatically during signup and stored using their UID as the document ID, making profile lookups extremely efficient (O(1) complexity).

#### 1.2 Listings Collection (`listings`)
**Purpose**: Store all service and place listings created by users.

**Document ID**: Auto-generated by Firestore

**Fields**:
```json
{
  "name": "string - Service or place name",
  "category": "string - Category enum value (hospital, restaurant, etc.)",
  "address": "string - Physical address",
  "contactNumber": "string - Phone number",
  "description": "string - Detailed description",
  "latitude": "number - Geographic coordinate",
  "longitude": "number - Geographic coordinate",
  "createdBy": "string - UID of creator",
  "timestamp": "Timestamp - Creation/update time"
}
```

**Indexes**:
- Composite index on `createdBy` (Ascending) + `timestamp` (Descending)
- Composite index on `category` (Ascending) + `timestamp` (Descending)

**Security Rules**:
- Read: Any authenticated user
- Create: Authenticated users only, must set their own UID as `createdBy`
- Update/Delete: Only the creator (UID matches `createdBy`)

**Rationale**: Using `createdBy` field enables efficient filtering of user-specific listings while maintaining a single collection for all listings, simplifying queries and reducing database complexity.

## 2. State Management Implementation

### Chosen Solution: Provider Pattern

**Justification**: Provider was selected for this project because:
1. **Clean Architecture**: Natural separation between business logic and UI
2. **Performance**: Fine-grained widget rebuilds via `Consumer` widgets
3. **Scalability**: Easy to add new providers as features grow
4. **Testing**: Business logic isolated in providers is easier to unit test
5. **Flutter Integration**: Official recommendation from Flutter team

### Provider Architecture

#### 2.1 AuthProvider
**Responsibility**: Manages authentication state and user session

**State Properties**:
- `status`: Current authentication status (authenticated, unauthenticated, emailNotVerified)
- `user`: Firebase User object
- `userProfile`: UserModel from Firestore
- `isLoading`: Loading state for async operations
- `errorMessage`: Error messages from auth operations

**Methods**:
- `signUp()`: Creates new user account with email verification
- `signIn()`: Authenticates existing user
- `signOut()`: Signs out current user
- `checkEmailVerified()`: Polls email verification status
- `resendVerificationEmail()`: Sends new verification email

**Data Flow**:
```
FirebaseAuth.authStateChanges 
  → AuthProvider._onAuthStateChanged()
  → Updates state
  → notifyListeners()
  → UI rebuilds via Consumer<AuthProvider>
```

#### 2.2 ListingProvider
**Responsibility**: Manages all listing data and filtering

**State Properties**:
- `listings`: All listings (filtered)
- `userListings`: Current user's listings
- `selectedCategory`: Active category filter
- `searchQuery`: Active search text
- `isLoading`: Loading state
- `errorMessage`: Error messages

**Methods**:
- `initializeListingsStream()`: Subscribes to all listings
- `initializeUserListingsStream()`: Subscribes to user's listings
- `createListing()`: Creates new listing
- `updateListing()`: Updates existing listing
- `deleteListing()`: Removes listing
- `setSearchQuery()`: Filters by name
- `setCategory()`: Filters by category
- `clearFilters()`: Resets all filters

**Data Flow**:
```
Firestore.collection('listings').snapshots()
  → ListingProvider stream listener
  → Applies filters (search + category)
  → Updates _filteredListings
  → notifyListeners()
  → UI rebuilds via Consumer<ListingProvider>
```

**Key Design Decision**: Real-time filtering is performed in the provider layer after receiving Firestore snapshots, rather than using Firestore queries. This allows combining search and category filters efficiently while maintaining real-time updates.

#### 2.3 SettingsProvider
**Responsibility**: Manages app settings and preferences

**State Properties**:
- `locationNotificationsEnabled`: Toggle state for notifications

**Methods**:
- `toggleLocationNotifications()`: Updates notification preference

## 3. Service Layer Design

### Separation of Concerns
Services handle direct Firebase/Firestore operations, while Providers manage state and business logic.

#### 3.1 AuthService
**Responsibilities**:
- Firebase Authentication operations
- User profile creation in Firestore
- Email verification management
- Error handling and message translation

**Key Method**: `signUp()`
```dart
1. Create user with Firebase Auth
2. Send verification email
3. Update display name
4. Create user profile document in Firestore
5. Return UserCredential
```

#### 3.2 ListingService
**Responsibilities**:
- CRUD operations on listings collection
- Stream-based data retrieval
- Query composition

**Key Design**: All read operations return `Stream<List<ListingModel>>` for real-time updates, while write operations return `Future<void>` for immediate feedback.

## 5. Feature Implementation Details

### 5.1 Search and Filtering
**Implementation**: Client-side filtering applied to Firestore stream results

**Process**:
1. Firestore streams all listings in real-time
2. Provider applies search filter (case-insensitive substring match)
3. Provider applies category filter (exact match)
4. Filtered results stored in `_filteredListings`
5. UI rebuilds automatically

**Trade-off**: Client-side filtering means all listings are loaded, which could be inefficient with thousands of listings. However, it enables:
- Real-time updates while filtering
- Combining multiple filter types
- No additional Firestore query costs

**Alternative Considered**: Firestore queries with `.where()` clauses, but this would require separate streams for each filter combination and doesn't support partial text matching.

### 5.2 Google Maps Integration

**Implementation**: Using `google_maps_flutter` package

**Features**:
- Embedded maps on detail pages
- Full-screen map view of all listings
- Custom marker colors by category
- Navigation to Google Maps for directions

**Key Code Pattern**:
```dart
GoogleMap(
  initialCameraPosition: CameraPosition(
    target: LatLng(latitude, longitude),
    zoom: 15,
  ),
  markers: {
    Marker(
      markerId: MarkerId(listing.id),
      position: LatLng(latitude, longitude),
    ),
  },
)
```

**Navigation**: Uses `url_launcher` package to open Google Maps with pre-filled destination, leveraging native Google Maps app for turn-by-turn directions.

### 5.3 Location Services

**Implementation**: Using `geolocator` package

**Features**:
- Request location permissions

- Auto-fill coordinates when creating listings

**Permission Handling**:

- Runtime permission requests handled by geolocator

## 6. Technical Challenges and Solutions

### Challenge 1: Email Verification Flow
**Problem**: Firebase Auth doesn't automatically redirect after email verification.

**Solution**: Implemented automatic polling in `EmailVerificationScreen` that:
- Checks verification status every 3 seconds
- Allows manual resend with 60-second cooldown
- Automatically transitions to authenticated state when verified

### Challenge 2: Real-time Filtering
**Problem**: Firestore doesn't support complex queries for partial text matches combined with category filters.

**Solution**: Stream all listings, apply filters client-side in Provider:
```dart
void _applyFilters() {
  _filteredListings = _listings.where((listing) {
    bool matchesSearch = _searchQuery.isEmpty ||
        listing.name.toLowerCase().contains(_searchQuery.toLowerCase());
    
    bool matchesCategory = _selectedCategory == null ||
        listing.category == _selectedCategory;
    
    return matchesSearch && matchesCategory;
  }).toList();
}
```

### Challenge 3: Marker Color Differentiation
**Problem**: Default Google Maps markers are all red.

**Solution**: Mapped categories to different hue values:
- Hospitals: Red
- Police Stations: Blue  
- Restaurants/Cafés: Orange
- Parks: Green
- Others: Violet

### Challenge 4: CRUD Operation Feedback
**Problem**: Users need immediate feedback for async operations.

**Solution**: Provider pattern with loading states:
```dart
Future<bool> createListing(ListingModel listing) async {
  _isLoading = true;
  notifyListeners(); // UI shows loading spinner
  
  try {
    await _listingService.createListing(listing);
    _isLoading = false;
    notifyListeners();
    return true;
  } catch (e) {
    _errorMessage = e.toString();
    _isLoading = false;
    notifyListeners();
    return false;
  }
}
```

## 7. Design Trade-offs

### Trade-off 1: Client-side vs Server-side Filtering
**Decision**: Client-side filtering in Provider

**Pros**:
- Real-time updates
- Complex filter combinations
- No query costs

**Cons**:
- All data loaded to device
- Doesn't scale to 10,000+ listings

**Justification**: For a city directory app, expecting hundreds (not thousands) of listings, client-side filtering provides better UX and simpler implementation.

### Trade-off 2: Single Collection vs Separate Collections per Category
**Decision**: Single `listings` collection with category field

**Pros**:
- Simpler queries
- Directory view shows all categories
- Easier to add new categories

**Cons**:
- Can't set different permissions per category

**Justification**: Permissions are based on creator, not category, so single collection is appropriate.

### Trade-off 3: State Management Pattern
**Decision**: Provider over Riverpod or BLoC

**Pros**:
- Simpler learning curve
- Less boilerplate
- Official Flutter recommendation

**Cons**:
- Less type-safe than Riverpod
- Less structured than BLoC

**Justification**: For this application size, Provider's simplicity outweighs benefits of more complex patterns. Can refactor to Riverpod later if needed.
